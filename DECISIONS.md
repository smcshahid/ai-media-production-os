### D-19 — Enum columns use portable `VARCHAR + CHECK` (`native_enum=False`), keys are app-side UUIDs
**Date:** 2026-06-09
**Decision:** Status/stage/decision/event-type columns are SQLAlchemy `Enum(..., native_enum=False)` (renders as `VARCHAR` + `CHECK` constraint, not a native PostgreSQL `ENUM` type). Primary keys are `uuid4` generated in the application; `created_at`/`updated_at` default to the DB clock (`func.now()`). A constraint **naming convention** is set on `Base.metadata` so the T-04-02 migration and all future autogenerate output have stable, reviewable names.
**Rationale:** Native PG enums are painful to migrate (every value add is a DDL dance); a checked VARCHAR keeps the enum contract while staying migration-friendly and portable (the importability test builds the schema on SQLite). App-side UUIDs avoid depending on `gen_random_uuid()` at insert time and keep the models DB-agnostic for unit tests. The enums themselves live in `packages/aimpos-core` (Sprint 0 plan §4.3) as the single source of truth shared by api and worker. Verified by `api/tests/unit/test_models_importable.py` (5 tests pass; `create_all` succeeds).

### D-20 — Migrations are generated from models; `make migrate` runs in a one-off container until the API image exists
**Date:** 2026-06-09
**Decision:** The initial Alembic migration (`0001_initial_core_tables`, T-04-02) is **autogenerated from the SQLAlchemy models** (T-04-01) against PostgreSQL and committed under `api/alembic/versions/`. `alembic/env.py` reads `DATABASE_URL` from the environment and sets `target_metadata = Base.metadata`. Because the API image does not exist until US-03/T-03-01, `make migrate` / `make migrate-down` run Alembic in a one-off `python:3.12-slim` container attached to the `aimpos-spark` network (installing the migration deps + editable `aimpos-core`); when the API image lands this becomes `docker compose run --rm api alembic <cmd>`.
**Rationale:** Generating from models keeps the migration and the ORM in lock-step (verified: Alembic `compare_metadata` reports zero drift). Reading `DATABASE_URL` directly in `env.py` is acceptable because migration tooling is not application code (the `os.getenv` ban in coding-standards §345 targets app code; settings still load via `aimpos-config` in US-03). The one-off container avoids adding a migration service to the Sprint-0 compose (no infra/scope change) while making `make migrate` work today; `DATABASE_URL`'s in-network host `postgresql:5432` resolves only inside the `aimpos-spark` network, which is why the command joins it. **Verification:** `upgrade head` creates all six tables, `downgrade base` removes them, re-`upgrade` succeeds, and the tables were confirmed in the live compose database.

### D-21 — Repositories: generic async SQLAlchemy adapter + Protocol port; caller owns the transaction
**Date:** 2026-06-09
**Decision:** T-04-03 adds one repository per aggregate root (`Project`, `PipelineRun`, `AssetVersion`, `Approval`, `AuditEvent`) under `api/app/infrastructure/db/repositories/`, built on a generic `SQLAlchemyRepository[ModelT]` and a `Repository` `Protocol` (the port). Repositories use `AsyncSession` and **`flush`, never `commit`** — the transaction boundary belongs to the caller (request scope / unit of work). Session plumbing (`session.py`) exposes pure `build_engine(url)` / `build_sessionmaker(engine)` builders with **no global engine and no env reads**; wiring to settings happens in US-03 via `aimpos-config`. `aimpos-core` ships a `py.typed` marker so its types flow to `api`/`worker` (mypy strict passes on `aimpos-core`; clean on `api`).
**Rationale:** A generic base keeps the five repositories tiny and consistent; the Protocol lets future domain/application services depend on an abstraction without importing SQLAlchemy (preserves domain purity, coding-standards §32). `flush`-not-`commit` avoids partial commits and lets `store_asset` (US-05) compose multiple writes atomically. `LineageEdge` gets no repository yet — it is a relation, not an aggregate root (added when US-14+ writes edges) — avoiding speculative scope. **Verification:** async round-trip test per repository on in-memory SQLite (`aiosqlite`) — 5/5 pass (10/10 with the model tests); `ruff` + `mypy` clean. This closes **US-04** (T-04-01/02/03 all done pending merge).

### D-23 — Request-id + structured access logs via pure-ASGI middleware; `request_id` flows through a context var (closes US-03)
**Date:** 2026-06-09
**Decision:** T-03-03 adds `RequestIDMiddleware` and T-03-02 adds `AccessLogMiddleware` under `api/app/middleware/` (Repository Structure §4.4). Both are **pure-ASGI** middleware (not Starlette `BaseHTTPMiddleware`) so the `request_id` `ContextVar` is set in the *same* context the endpoint runs in. The id is taken from an inbound **`X-Request-ID`** header when present (cross-service correlation), else a UUID4 is generated; it is always echoed on the response. The contextvar (`request_id_var`) and a `RequestIdFilter` live in **`aimpos-config`** so the same correlation mechanism is reusable by the worker; the filter injects `request_id` into every `LogRecord`, so *all* logs emitted during a request (access line, httpx, handler) share the id. `AccessLogMiddleware` is outermost-but-one (RequestID wraps it) and logs one JSON line per request (`http_method`, `path`, `status_code`, `duration_ms`, `client`, `request_id`). Uvicorn's default plaintext access log is disabled (`--no-access-log`) since the middleware supersedes it.
**Rationale:** `BaseHTTPMiddleware` runs the downstream app in a child task whose contextvar writes don't reliably propagate; pure-ASGI avoids that class of bug and is lower overhead. Honouring an inbound `X-Request-ID` lets a proxy/caller (and, later, Temporal/agent spans) correlate across the call chain — aligning with the `correlation_id` in the audit/knowledge-graph models (Technology Recommendations §4.8) without yet pulling in the full OpenTelemetry stack. Keeping `request_id_var` in `aimpos-config` (not the api) prevents the worker from re-inventing it. This satisfies the remaining US-03 AC ("structured JSON logs with `request_id`"); the `temporal` entry in the US-03 story AC is **Sprint 1** (Sprint 0 plan §4.6), so US-03 is functionally complete for Sprint 0.
**Verification:** 21/21 tests pass (request-id generated/echoed/distinct; `JsonFormatter`+`RequestIdFilter` unit; end-to-end access-log correlation); `ruff` + `mypy` (strict on `aimpos-config`) clean. **Live stack:** an inbound `X-Request-ID: smoke-trace-9999` appeared on the response header **and** on both the `aimpos.access` JSON line and the in-handler `httpx` MinIO-probe line — proving request-scoped correlation; generated requests get distinct UUID4 ids; no duplicate plaintext access line after `--no-access-log`. **Known minor gap (TD-19):** uvicorn's two startup banner lines remain plaintext (separate logger, `propagate=False`).

### D-25 — US-05: `store_asset` is a ports-&-adapters application service in `api/domain/assets/`; SHA-256 content hash, MD5/ETag verified after upload; blob dedup with per-chain version rows (closes US-05)
**Date:** 2026-06-09
**Decision:** US-05 delivers the **asset storage service**, not an HTTP route. (a) **T-05-02** — pure `api/app/domain/assets/content.py`: `compute_content_hash(bytes) -> sha256-hex` and `build_object_key(project_id, stage, content_hash) -> "{project_id}/{stage}/{content_hash}"` (deterministic). (b) **T-05-03** — `api/app/domain/assets/service.py` defines the framework-free `store_asset(...)` plus two **ports** (`BlobStore`, `AssetVersionStore` `Protocol`s) and a `StoredAsset` DTO; it hashes → uploads via `BlobStore` → `next_version` → `add_version`, returning `StoredAsset`. The infrastructure `AssetVersionRepository.add_version(...)` maps the ORM row to `StoredAsset` (port impl). (c) **T-05-01** — `api/app/infrastructure/storage/minio_client.py`: `MinioClient` (implements `BlobStore`) wrapping the **sync** `minio` SDK with `asyncio.to_thread`; `upload_bytes`/`download_bytes`/`stat_object`; typed `StorageError`/`ObjectNotFoundError`; reads endpoint/keys/bucket from `aimpos-config` `Settings`. (d) **T-05-04** — `api/tests/integration/test_asset_upload.py`, marked `integration` and skipped unless `AIMPOS_RUN_INTEGRATION=1`, exercises the full round-trip + dedup against compose MinIO + PostgreSQL.
**Rationale:** Repository Structure §"Asset Service → api/domain/assets/" places the service in the domain, while coding-standards §32 bans SQLAlchemy/HTTP-client imports there — **ports & adapters** is the only design that satisfies both: the domain depends on `Protocol`s, the adapters (MinIO client, repository) are injected by the caller. This reuses the `flush`-not-`commit` repository (D-21) so a future `POST /assets` can compose the blob+row writes in one transaction. **content_hash vs ETag:** the US-05 AC says "content_hash matches ETag", but MinIO's single-PUT **ETag is the body MD5**, not SHA-256; we record SHA-256 as the content address (the dedup key) and satisfy the AC's "verified after upload" by having `upload_bytes` re-check the returned ETag against `md5(data)` and raise `StorageError` on mismatch. **Dedup policy (per AC + MVP):** identical bytes for a `(project, stage)` reuse the same content-addressed **key** (one blob) but always append a **new version row** (`next_version` increments along the `(project_id, stage)` chain, guarded by the existing unique constraint).
**Scope guard / planning gap:** US-05's four tasks are the **service only** — there is **no `POST/GET /assets` route task** in US-05 (unlike US-01's `T-01-03`). The "Upload Asset" Sprint-0 success criterion needs that HTTP surface, but no Sprint-0 task defines it; recorded as **TD-22** (add the assets route before the Week-3 frontend integration) rather than inventing scope here. `pipeline_run_id` is left `NULL` for human uploads (the column is nullable by design, T-04-01).
**Verification:** 33/33 unit tests pass (3 content + 3 `store_asset`/dedup added), integration skipped by default; `ruff` + `mypy` clean. **Live stack:** `AIMPOS_RUN_INTEGRATION=1` against compose MinIO+PostgreSQL — `store_asset` of identical bytes twice produced versions **1** and **2** with the **same** `minio_key` and `content_hash`, the object **downloaded byte-identical**, and the downloaded SHA-256 matched the recorded `content_hash`. (Windows-host note: an integration `conftest.py` selects `WindowsSelectorEventLoopPolicy` because `psycopg` async rejects the default `ProactorEventLoop`; no-op on Linux/CI.)

### D-26 — `POST /assets` + `GET /assets` HTTP surface (resolves TD-22); thin controllers own the transaction, human uploads are `is_ai_generated=False`
**Date:** 2026-06-09
**Decision:** Add the asset HTTP endpoints the Sprint-0 plan requires but no task had defined (the gap recorded as TD-22 in D-25): **`POST /assets`** (Sprint 0 plan §4.6) and **`GET /assets`** (§4.7). They live in `api/app/routes/assets.py` as **thin controllers**: `POST` accepts a `multipart/form-data` body (`project_id`, `stage`, `file`), validates the project exists (**404** if not — *before* any MinIO write, to avoid orphan blobs), calls the framework-free `store_asset` (D-25) with the injected `MinioClient` (`BlobStore`) + `AssetVersionRepository` (`AssetVersionStore`), **commits** (the route is the unit-of-work boundary; repositories only `flush`), and returns **201** with the created `AssetRead`. `GET /assets?project_id=…` returns the project's versions ordered `created_at` **DESC** (`list_for_project`). `MinioClient` is created once in the app lifespan (`app.state.minio`) and exposed via a `get_minio` provider; `python-multipart` is added for form parsing. Storage failures surface as **502**.
**Rationale:** §4.7's DoD explicitly lists `GET /assets`, and §4.6 lists `POST /assets` as a Basic-Backend endpoint — so this is **completing US-05's Asset-Storage component**, not new scope (US-05 had under-delivered by shipping only the service). `POST /assets` is the **human** upload path, so `is_ai_generated` is hard-wired **False** (plan §4.7); agent outputs set `True` by calling `store_asset` from the worker, never this route — so the flag is deliberately **not** a request field. Project-existence is checked before upload so a bad id can't leave an orphan object in MinIO. **Compensation note:** `store_asset` writes the blob before the row flush; on a later DB failure the content-addressed object is left in MinIO, but because the key is the SHA-256 it is **benign and self-deduplicating** (a retry reuses it) — true delete-on-failure compensation is deferred (recorded as TD-25), matching the plan's "atomic … or compensates" allowance for MVP. Auth is **not** added here — these mirror `GET /projects` and inherit the Bearer guard with **US-25**.

### D-09 — API auth is a single Bearer token from `AIMPOS_API_TOKEN`; Keycloak deferred to Phase 1 (US-25 backend)
**Date:** 2026-06-09
*(Recorded here per the append-only policy; the ID matches the Sprint 0 plan §6 "D-09" the §4.5 DoD asks for. US-25 is **S2** in the old issue tracker but a **Sprint-0** deliverable per the higher-authority Sprint 0 plan §4.5 + the exit gate "auth middleware live on all routes" — a Sprint reclassification.)*
**Decision:** API authentication for the MVP is a **single shared static Bearer token** read from `AIMPOS_API_TOKEN` (via `aimpos-config` `Settings.api_token`). `api/app/middleware/auth.py` adds a **pure-ASGI** `AuthMiddleware` that rejects every HTTP request **except `GET /health`** with **401 `{"detail": "Unauthorized"}`** (plus `WWW-Authenticate: Bearer`) unless the request carries `Authorization: Bearer <token>` matching the configured value (constant-time `secrets.compare_digest`; an empty configured token fails closed). This is a deliberate, slightly stricter reading than "keep docs open": it avoids unauthenticated schema disclosure and keeps the exemption list trivially auditable; the schema is still reachable with the token (`curl -H "Authorization: Bearer …" /openapi.json`). **Not yet handled — CORS/OPTIONS:** the cross-origin frontend's credential-less preflight `OPTIONS` will be 401'd by this middleware; a `CORSMiddleware` (outermost) + preflight handling lands with US-26 (recorded as TD-26).

### D-44 — US-16 — Storyboard batch completeness contract
**Date:** 2026-06-11
**Decision:** A **successful storyboard batch** is defined as a single atomic outcome of one `run_storyboard_agent` activity attempt where **all** of the following hold: (1) the Cinematography planner returns **`4 <= frame_count <= 6`** unique shot prompts; (2) **every** shot is rendered to valid PNG bytes via ComfyUI; (3) **every** PNG passes `validate_storyboard_frame()` (magic bytes / decode check, non-empty); (4) **all N frames** are persisted to MinIO and **all N** `asset_versions` rows are inserted in **one database transaction** sharing the batch `version` from `D-43`; (5) **all N** `lineage_edges` rows (script parent → each frame child) are inserted in the same transaction; (6) **all N** `ASSET_STORED` audit events are appended after commit. **Partial batches are forbidden:** if **any** frame fails during planning, GPU sequencing, ComfyUI queue/poll, validation, MinIO write, DB insert, or lineage insert, the activity **MUST NOT** leave any STORYBOARD `asset_versions` rows, MinIO objects, or lineage edges from that attempt. The activity raises, Temporal retries the **entire** activity (workflow `RetryPolicy(maximum_attempts=2)` per Issue 38 AC-5), and only after exhausted retries does `_mark_storyboard_failed` set `pipeline_runs.status=FAILED` with `PIPELINE_FAILED` audit. **No compensating deletes** of prior successful batches — failed attempts are simply absent from the chain; a later successful attempt appends a new batch `version`.
**Rationale:** Visual MVP AC-1 and AC-3 require a coherent 4–6 frame set linked to the approved script. Allowing orphan frames would break US-17 "approve all" semantics and corrupt lineage. All-or-nothing persistence matches the single Temporal activity boundary and keeps retry semantics simple.
**Verification:** US-16 unit tests — simulated mid-batch ComfyUI failure asserts zero STORYBOARD rows; Olares happy path shows contiguous `frame_index` 1..N at single `version`; optional negative test confirms `FAILED` after retry exhaustion with no partial rows.
