# Scope Change Request — Character Bible Pilot

**SCR ID:** SCR-2026-005

## Summary

Authorize bounded **character continuity** across scenes and episodes: structured Character entities (max 3 per project), character bible injection into story/script/storyboard generation, **export manifest v5** when characters are selected, with full backward compatibility for manifest v1–v4.

---

## Definitions

| Term | Definition |
|------|------------|
| **Character** | Project-scoped entity with name, description, role, visual traits, personality notes (Alembic 0006 `characters` table). |
| **Character Profile** | The five pilot attributes stored on a Character row; no dynamic memory or embeddings. |
| **Character Continuity** | Selected characters are bound to a pipeline run (`pipeline_runs.character_ids`) and their formatted bible is injected into STORY, SCRIPT, and STORYBOARD agent prompts to reduce cross-scene/episode drift. |

---

## Scope

| In scope | Out of scope |
|----------|--------------|
| Character CRUD (max 3/project) | Character memory / chat / agents |
| Run binding via `character_ids` | Vector DB, Neo4j, RAG |
| Worker bible injection | Voice cloning, music |
| Manifest v5 + v1–v4 compat | OAuth, RBAC, collaboration |
| Simple dashboard UX | Studio workflows, asset restore |

---

## Architecture

```
POST /characters → characters table
POST /pipeline/start { character_ids[] } → pipeline_runs.character_ids
Worker: fetch profiles → format_character_bible() → STORY / SCRIPT / STORYBOARD prompts
Export: manifest v5 when characters present; v4 ladder unchanged otherwise
```

**Stop conditions evaluated:** No graph/vector infra; episodes/lineage/history/audit unchanged; pilot capped at 3 characters.
