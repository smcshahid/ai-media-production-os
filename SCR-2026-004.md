# Scope Change Request — Episode Model Pilot

**SCR ID:** SCR-2026-004  
**Date:** 2026-06-17  
**Status:** **ACCEPTED — Phase 6 Episode Model Pilot authorized**  
**Baseline release:** `v0.15.0-phase5-narration`  
**Authority:** Phase 6 mission charter · US-V06 accepted

---

## Summary

Evolve AIMPOS from a scene generator into an **episode-oriented creator platform** by introducing an Episode entity between Project and Scene, while preserving all governance, lineage, history, audit, export, and release guarantees established through Phase 5.

---

## Domain model

```
Project
  └── Episode (1..N per project, pilot: 1–2)
        └── Scene (1..3 per episode, existing multi-scene pilot)
              └── Assets (IDEA shared at project; STORY→VIDEO per run)
```

| Entity | Responsibility |
|--------|----------------|
| **Project** | Organizational shell; shared IDEA; episode container |
| **Episode** | Numbered production unit; own pipeline run; export scope |
| **Scene** | Fountain block within episode script (Phase 4 model preserved) |
| **Assets** | Versioned bytes; scoped by `pipeline_run_id` + metadata |

---

## Lifecycle model

| Phase | Episode status | Trigger |
|-------|----------------|---------|
| Created | `DRAFT` | `POST /episodes` |
| Pipeline active | `IN_PROGRESS` | `POST /pipeline/start` with `episode_id` |
| Run completed | `COMPLETED` | Worker sync → COMPLETED |
| Archived | `ARCHIVED` | Future operator action (out of pilot scope) |

Legacy runs without `episode_id` remain **project-scoped** with unchanged lifecycle.

## Scope

| In scope | Out of scope |

| Episode entity + schema | Character bible |

---

## Governance impact analysis

| System | Impact | Redesign required? |
|--------|--------|-------------------|
| **Lineage** | `episode_number` in node metadata | **No** — append metadata |
| **History** | Sort key adds `episode_number` | **No** — extend sort |
| **Audit** | `episode_id` in PIPELINE_STARTED payload | **No** — append payload |
| **Export** | Manifest v4 + `episodes/episode_XX/` paths | **No** — version ladder |
| **Realtime** | Status push includes episode fields | **No** — extend D-59 model |
| **Run history** | Summary includes `episode_id` | **No** — additive fields |

**Stop conditions evaluated:** No lineage/history/audit redesign; no destructive migration; multi-scene preserved; manifest backward compat via version ladder.

---

## Migration strategy

1. **Alembic 0005** (additive only):
   - `episodes` table
   - `pipeline_runs.episode_id` nullable FK
2. Existing rows: `episode_id = NULL` → legacy behavior unchanged
3. No data backfill required
4. Export: v1/v2/v3 when `episode_id IS NULL`; v4 when set

---

## Architecture

```
POST /episodes → Episode(DRAFT)
POST /pipeline/start { episode_id, scene_count }
  → PipelineRun(episode_id) → SparkPipelineWorkflow
  → Assets(pipeline_run_id, metadata.episode_number)
GET /export/{run_id} → manifest v4 (episode paths)
```
